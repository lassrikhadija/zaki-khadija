Core Web Vitals 2026 : comment savoir si votre site est assez rapide — et quoi corriger en premier

Khadija AitLassri 28 avril 2026 9 min de lecture

Résumé en 30 secondes

  • Les Core Web Vitals, c'est 3 chiffres. LCP (temps avant que la page s'affiche), INP (temps avant qu'elle réagisse à un clic), CLS (stabilité visuelle au chargement). Google les mesure, les publie et les utilise pour classer.
  • Les seuils à connaître : LCP ≤ 2,5 s · INP ≤ 200 ms · CLS ≤ 0,1. En 2026, un site qui rate ces seuils perd du trafic organique de manière mesurable.
  • Le test dure 2 minutes : PageSpeed Insights, gratuit — vous tapez votre URL, vous lisez 3 chiffres. Rouge = urgence. Orange = à corriger. Vert = vous passez l'examen.
  • Les 3 causes les plus fréquentes : images non compressées, scripts tiers envahissants, hébergement mutualisé bas de gamme. Ces 3 corrections coûtent entre 0 et 300 $ et donnent l'essentiel du gain.

Les 3 métriques Core Web Vitals qui décident

Les Core Web Vitals (CWV) sont les 3 indicateurs que Google utilise pour mesurer l'expérience utilisateur technique de votre site. Ils sont officiellement intégrés aux facteurs de classement depuis juin 2021 et renforcés en mars 2024 avec le remplacement de FID par INP. Source : Google Search Central — Core Web Vitals.

Les 3 métriques mesurent 3 choses concrètes que ressent votre visiteur :

  • LCP — Largest Contentful Paint. Combien de temps avant que l'élément principal de votre page (image hero, titre principal) s'affiche ? Ressenti utilisateur : « Ça charge ? »
  • INP — Interaction to Next Paint. Combien de temps entre un clic ou tap de l'utilisateur et la première réaction visible de la page ? Ressenti : « Le bouton réagit, oui ou non ? »
  • CLS — Cumulative Layout Shift. À quel point vos éléments bougent pendant le chargement ? Un CLS élevé, c'est l'expérience « je clique sur un lien, mais il se décale au dernier moment et j'ouvre autre chose ».
Le saviez-vous ? Selon le Chrome User Experience Report 2025, 52 % des sites web mondiaux échouent sur au moins une des 3 Core Web Vitals. Pour les PME, la proportion est encore plus élevée — la performance technique est souvent le premier levier sacrifié par manque de temps.

Les seuils Google 2026 (et pourquoi 2,5 s n'est pas un objectif marketing)

Google définit 3 paliers : Bon (vert), À améliorer (orange), Mauvais (rouge). Ce ne sont pas des recommandations : ce sont les seuils exacts utilisés par l'algorithme de classement.

Métrique Bon (vert) À améliorer (orange) Mauvais (rouge)
LCP ≤ 2,5 s 2,5 – 4,0 s > 4,0 s
INP ≤ 200 ms 200 – 500 ms > 500 ms
CLS ≤ 0,1 0,1 – 0,25 > 0,25

Sources : Google — web.dev/vitals · Chrome for Developers

Attention : Google utilise les mesures sur utilisateurs réels (Chrome UX Report), pas celles faites en laboratoire. Votre propre ordinateur avec fibre gigabit ne reflète pas le ressenti de votre client sur 4G à Trois-Rivières.

Le saviez-vous ? Deloitte — Milliseconds Make Millions : une amélioration de 0,1 s du LCP entraîne en moyenne +8 % de conversion e-commerce et +10 % de temps passé sur les pages lead-gen. Ce n'est pas un détail technique — c'est un levier de chiffre d'affaires.

Comment tester votre site en 2 minutes (PageSpeed Insights)

Vous n'avez besoin d'aucun outil payant pour diagnostiquer vos Core Web Vitals. Procédure en 4 étapes :

  1. Allez sur pagespeed.web.dev.
  2. Collez l'URL de votre page d'accueil. Cliquez « Analyser ».
  3. Attendez 30-60 secondes.
  4. Lisez les 3 chiffres dans la section « Découvrez les performances de vos utilisateurs réels ».

Vous verrez une carte mobile et une carte bureau. Concentrez-vous sur mobile : la majorité du trafic web québécois passe sur mobile, et Google indexe en mobile-first depuis 2020. Source : Statistique Canada — Enquête Internet.

Si vos 3 chiffres sont verts sur mobile, votre site passe l'examen CWV. Si un seul est rouge ou orange, vous avez un chantier. Deuxième test utile : faites-le sur votre page de conversion (contact, audit, devis) — c'est souvent celle qui traîne davantage.

Les 5 causes de lenteur les plus courantes sur les sites PME

1. Images non optimisées

Des JPG à 2 Mo livrés non compressés, non dimensionnés pour l'affichage réel, chargés au-dessus de la ligne de flottaison sans attribut fetchpriority. Impact : LCP qui dérape facilement à 4-6 s. Correction : Squoosh ou TinyPNG, conversion WebP, redimensionnement. Coût : 0 $.

2. Scripts tiers envahissants

Chat Messenger, pixel Facebook, Google Analytics, Hotjar, trackers SEO, plugin popup exit. Chaque script tiers est une boîte noire qui peut bloquer le rendu. Selon le HTTP Archive — Web Almanac 2024, les scripts tiers représentent en médiane 45 % du temps de chargement d'une page. Correction : audit des scripts, suppression de ceux inutilisés, chargement différé (async/defer) des autres.

3. Hébergement mutualisé bas de gamme

Un hébergement à 4 $/mois partagé avec des centaines d'autres sites donne un TTFB (time to first byte) de 1-2 secondes, ce qui plombe tout le reste. Correction : passer à un hébergement VPS ou managé (Kinsta, WP Engine, Pressable, Cloudways) à partir de 30 $/mois.

4. Thème ou constructeur lourd

Les thèmes pagebuilder avec toutes les options activées peuvent générer plusieurs Mo de CSS et JS par page. Un thème léger (GeneratePress, Kadence, Astra) reste bien en dessous. Correction : changement de thème — chantier significatif, à planifier.

5. Mises en page qui sautent pendant le chargement

Bannières, images sans dimensions fixes, polices web chargées tardivement. Impact : CLS élevé. Correction : déclarer width et height sur toutes les images, précharger les polices critiques avec rel="preload".

Diagnostiquer prend 2 minutes. Prioriser les corrections prend 30 minutes. Appliquer les corrections prend entre 2 h et 40 h selon la dette technique. Vous voulez un second regard ?

Le plan de correction en 30 jours

Semaine 1 — Le dirigeant (2 h)

  • Tester la home et les 5 pages les plus visitées dans PageSpeed Insights.
  • Noter les 3 chiffres par page dans un tableau simple.
  • Identifier laquelle des 5 causes (images, scripts, hébergement, thème, CLS) est la principale.

Semaine 2 — Délégué (développeur ou agence, 8-12 h)

  • Compression et conversion WebP de toutes les images de plus de 200 Ko.
  • Audit des scripts tiers, suppression de ceux sans usage réel.
  • Ajout de async/defer sur les scripts non critiques.

Semaine 3 — Délégué (4-8 h)

  • Vérification de l'hébergement. Si TTFB > 800 ms, migration planifiée.
  • Ajout des dimensions width/height sur toutes les images.
  • Préchargement des polices critiques.

Semaine 4 — Le dirigeant (1 h)

  • Re-test PageSpeed Insights sur les mêmes pages.
  • Validation des gains. Mesure des impressions et du CTR dans Search Console.

Budget typique : 800 $ à 3 500 $ pour une PME de 20-100 pages selon la dette technique. Si on vous demande 8 000 $ et plus sans vous avoir montré vos 3 chiffres actuels, demandez un second devis.

Comment vérifier que vos corrections fonctionnent

Les données CrUX sont mises à jour tous les 28 jours. Un vrai verdict « Google » prend donc environ 4 semaines. Mais vous avez des signaux intermédiaires plus rapides :

  • PageSpeed Insights onglet « Diagnostics » : mesure lab en direct, utile pour valider qu'une correction fonctionne techniquement.
  • WebPageTest : test approfondi avec cascade de chargement, identifie si un script précis ralentit tout.
  • Search Console — Rapport Core Web Vitals : Expérience → Core Web Vitals. Mise à jour hebdomadaire.

Le signal CTR dans Search Console bouge en 2-3 semaines après une amélioration de vitesse — les pages plus rapides ont un meilleur CTR mobile à position égale.

Le saviez-vous ? Selon Google Search Central — Page Experience (2024), le passage d'un LCP « mauvais » à « bon » améliore le trafic organique de manière significative sur 60-90 jours, toutes choses égales par ailleurs.
