Schema Dentist : le balisage invisible qui fait citer votre clinique par ChatGPT et Google AI Overviews

Comment les IA lisent votre site (HTML vs données structurées)

Votre site web contient deux types d'informations. D'abord, le HTML visible : le texte que les visiteurs lisent, les images, les menus. Ensuite, les données structurées : un code technique ajouté dans les coulisses, invisible pour les humains, mais conçu spécifiquement pour être lu et compris par les moteurs de recherche et les intelligences artificielles.

La différence est fondamentale. Quand Google ou ChatGPT explore un site dentaire sans données structurées, il doit inférer les informations : il tente de deviner votre adresse en lisant votre page Contact, vos horaires en scannant du texte en français, vos services en analysant les titres de vos pages. Ce processus est imprécis. L'IA peut se tromper, rater des informations ou ignorer le cabinet faute de confiance dans les données extraites.

Avec le Schema Dentist JSON-LD, vous lui remettez directement une fiche structurée : « Voici mon nom exact, voici mon adresse en format standardisé, voici mes horaires du lundi au vendredi, voici mes services avec leurs noms officiels. » L'IA n'a plus à inférer — elle lit. Et ce qu'elle lit clairement, elle peut le citer avec confiance.

Pourquoi JSON-LD et pas Microdata ? Google recommande officiellement le format JSON-LD (JavaScript Object Notation for Linked Data) pour toutes les données structurées. Il s'insère dans une balise <script type="application/ld+json"> sans modifier le HTML visible de vos pages. C'est la méthode la plus simple à implémenter, à maintenir, et à valider.

Qu'est-ce que Schema.org type Dentist — l'héritage MedicalBusiness

Schema.org est un vocabulaire commun créé et maintenu par Google, Microsoft, Yahoo et Yandex. Il définit des centaines de types d'entités — Restaurant, Hotel, Person, Product — avec leurs propriétés standardisées. Pour les cabinets dentaires, le type officiel est Dentist.

Ce type s'inscrit dans une hiérarchie :

  • Thing → Organization → LocalBusiness → HealthAndBeautyBusiness ou MedicalBusiness → Dentist

Cette hiérarchie est importante : le type Dentist hérite de toutes les propriétés des types parents. Cela signifie que vous pouvez utiliser les propriétés de LocalBusiness (adresse, horaires, téléphone, géolocalisation) et les propriétés de MedicalBusiness (spécialité médicale, langues parlées, services disponibles) dans un seul bloc JSON-LD cohérent.

Pour les IA génératives, cette richesse est précieuse. Une IA qui reçoit la requête « Dentiste bilingue français-anglais spécialisé en orthodontie à Ville-Émard » peut croiser plusieurs propriétés structurées pour identifier les cabinets correspondants — mais seulement si ces propriétés existent dans les données.

Les propriétés obligatoires — ce que Google exige

Un Schema Dentist minimal doit comporter les propriétés suivantes pour être valide aux yeux de Google et pour déclencher des améliorations dans les résultats de recherche :

Propriété Type Description Statut
@type Text Toujours "Dentist" Requis
name Text Nom exact de la clinique tel qu'il apparaît sur Google Maps et GBP Requis
address PostalAddress Adresse complète — numéro, rue, ville, province, code postal, pays Requis
telephone Text Numéro avec indicatif régional, format E.164 recommandé Requis
url URL URL canonique de votre site Requis
openingHours Text Horaires au format standard Schema (ex. Mo-Fr 08:00-17:00) Requis
geo GeoCoordinates Latitude et longitude exactes (depuis Google Maps) Recommandé
Cohérence NAP absolue Le name, l'address et le telephone dans votre Schema doivent être strictement identiques à ce qui figure sur votre fiche Google Business Profile, votre page Contact et l'ensemble de vos répertoires locaux. Une incohérence — même mineure (« Avenue » vs « Ave. ») — réduit la confiance de Google dans vos données structurées.

Les propriétés avancées pour maximiser la citabilité IA

C'est ici que la majorité des cabinets dentaires laissent de la visibilité sur la table. Les propriétés avancées ne sont pas techniquement requises par Google pour valider le Schema — mais elles sont déterminantes pour être cité par les IA génératives, qui s'en servent pour répondre aux requêtes précises des patients.

Propriété Description Impact IA
medicalSpecialty Spécialité dentaire (ex. Orthodontie, Endodontie, Pédiatrie) Très élevé
availableService Liste des services — détartrage, blanchiment, implant, urgence dentaire, etc. Très élevé
knowsLanguage Langues parlées par l'équipe (français, anglais, espagnol, arabe…) Très élevé
priceRange Fourchette de prix ($, $$, $$$) — information attendue par les IA Élevé
paymentAccepted Modes de paiement (carte de crédit, débit, assurances) Élevé
hasMap URL Google Maps de votre fiche Moyen
sameAs URLs de vos profils GBP, Facebook, Instagram, Pages Jaunes Élevé
potentialAction Lien direct de prise de rendez-vous (ReserveAction) Très élevé
aggregateRating Note moyenne et nombre d'avis (source externe requise) Très élevé

La propriété availableService mérite une attention particulière. Chaque service que vous listez est un terme sur lequel une IA peut vous citer. Si un patient demande à ChatGPT « Qui fait des implants dentaires à Villeray ? », seuls les cabinets ayant explicitement déclaré "name": "Implants dentaires" dans leur Schema seront identifiés avec certitude.

Comment trouver votre CID Google Maps Recherchez votre clinique sur Google Maps, cliquez sur votre fiche, et regardez l'URL — elle contient un paramètre cid=XXXXXXXXXXXXXXX. C'est votre identifiant unique Google. Intégrez-le dans les champs hasMap et sameAs pour renforcer la connexion entre votre Schema et votre fiche GBP.

Où placer le script dans votre HTML

La recommandation officielle de Google est de placer le JSON-LD dans la balise <head> de votre page. Techniquement, il fonctionne aussi avant la fermeture </body>, mais la pratique conseillée reste le <head> pour une détection plus rapide lors du crawl.

Sur une seule page ou sur tout le site ?

Le Schema Dentist doit idéalement être présent sur toutes les pages de votre site, pas seulement la page d'accueil. Google associe les données structurées à la page sur laquelle elles se trouvent. Si un patient atterrit directement sur votre page « Implants dentaires » depuis une recherche IA, le Schema sur cette page confirme immédiatement que vous êtes bien la clinique concernée. Sur une page spécifique à un service, vous pouvez ajouter un Schema MedicalProcedure en complément du Schema Dentist global.

Attention aux doublons

N'incluez qu'un seul bloc Schema Dentist par page. Si vous utilisez un plugin WordPress comme RankMath ou Yoast qui génère automatiquement des données structurées, vérifiez qu'il ne crée pas déjà un Schema LocalBusiness pour éviter les conflits. Consultez le code source de votre page (Ctrl+U) et cherchez application/ld+json pour voir ce qui est déjà présent.

Valider avec Google Rich Results Test et Schema Markup Validator

Une fois le Schema en place, validez-le immédiatement avec ces deux outils officiels gratuits :

1 — Google Rich Results Test

Rendez-vous sur search.google.com/test/rich-results, entrez l'URL de votre page, et lancez le test. Google indique si votre Schema est valide, s'il peut générer des résultats enrichis (rich results), et liste toutes les erreurs ou avertissements avec leur description précise. C'est l'outil de référence pour un feedback immédiat.

2 — Schema Markup Validator

Rendez-vous sur validator.schema.org pour une validation technique complète. Cet outil vérifie la conformité de votre JSON-LD aux spécifications officielles Schema.org, indépendamment de Google. Il est particulièrement utile pour détecter les erreurs de syntaxe JSON ou les propriétés mal typées.

Erreur fréquente à la validation La plupart des erreurs Schema Dentist proviennent de trois sources : les guillemets incorrects (guillemets typographiques « » à la place de " "), les virgules manquantes entre les propriétés, et les URL mal formatées (manque du protocole https://). Validez toujours votre JSON dans un validateur JSON standard (jsonlint.com) avant de le soumettre au Google Rich Results Test.

Schema FAQPage — maximiser la citabilité IA avec une FAQ structurée

Le Schema FAQPage est le complément idéal du Schema Dentist. Il permet de déclarer formellement les questions fréquentes de vos patients — avec leurs réponses — dans un format que les IA lisent directement. C'est le mécanisme le plus efficace pour apparaître dans les réponses aux questions conversationnelles IA.

Quand ChatGPT ou Google AI Overviews reçoit une question comme « Est-ce que les implants dentaires font mal ? », il cherche des sources structurées qui ont répondu explicitement à cette question. Un cabinet avec un Schema FAQPage couvrant 10 questions de ce type est nettement mieux positionné pour être cité — et cité précisément — qu'un cabinet dont le contenu FAQ n'est présent qu'en HTML.

JSON-LD — Schema FAQPage (ajouter après le Schema Dentist)

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "Acceptez-vous les assurances dentaires au Québec ?",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "Oui. Nous travaillons avec la majorité des assureurs québécois incluant La Capitale, Sun Life, Great-West, Desjardins et Industrielle Alliance. Nous effectuons la facturation directe à votre assureur pour les soins couverts."
      }
    },
    {
      "@type": "Question",
      "name": "Offrez-vous des rendez-vous d'urgence dentaire le jour même ?",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "Oui. Nous réservons des plages horaires quotidiennes pour les urgences dentaires — douleur aiguë, dent cassée, couronne décollée. Appelez-nous le matin et nous vous recevons généralement dans la journée."
      }
    },
    {
      "@type": "Question",
      "name": "Parlez-vous anglais à la clinique ?",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "Oui. Toute notre équipe est bilingue français-anglais. Certains membres parlent également l'arabe et l'espagnol."
      }
    }
  ]
}
</script>

Adaptez les questions à votre réalité clinique. Les meilleures questions pour la citabilité IA sont celles qui correspondent aux requêtes conversationnelles réelles de vos patients : langues parlées, assurances acceptées, disponibilité urgences, fourchette de prix, spécialités.

Vérifier dans Google Search Console

Une fois votre Schema validé et votre site crawlé par Google (généralement quelques jours à quelques semaines), vous pouvez suivre son statut dans Google Search Console. Dans le menu de gauche, section « Améliorations », vous trouverez les entrées liées à vos données structurées.

GSC vous indique : combien de pages ont été détectées avec des données structurées valides, lesquelles présentent des erreurs ou avertissements, et dans certains cas, si vos données structurées ont déclenché des rich results dans les résultats de recherche. Un Schema Dentist valide sans erreur dans GSC est le signe que Google a correctement compris et indexé les informations de votre clinique.

Soumettez également une nouvelle demande d'indexation via « Inspection d'URL » après avoir ajouté ou modifié votre Schema — cela accélère la prise en compte des changements.

Erreurs fréquentes à éviter

Ces erreurs sont observées dans la majorité des cabinets dentaires qui tentent d'implémenter le Schema sans accompagnement :

Adresse incohérente (NAP)

C'est l'erreur la plus commune et la plus dommageable. Si votre fiche GBP dit « 1250, avenue du Mont-Royal Est » et votre Schema dit « 1250 Avenue Mont-Royal E », Google traite les deux comme des entités potentiellement différentes. Choisissez un format unique et appliquez-le sur tous vos points de présence en ligne sans exception.

Horaires mal formatés

Le format Schema standard pour les horaires est strict : "Mo-Fr 09:00-17:00". Les abréviations anglaises des jours sont obligatoires (Mo, Tu, We, Th, Fr, Sa, Su). Un format comme "Lundi-Vendredi 9h-17h" ne sera pas reconnu. Utilisez le guide officiel Google pour le formatage des horaires.

Propriétés manquantes critiques

Un Schema sans geo (coordonnées GPS) est moins performant pour les requêtes géolocalisées. Un Schema sans availableService ne peut pas être cité pour des requêtes de services spécifiques. Un Schema sans knowsLanguage passe à côté de toutes les requêtes bilingues. Chaque propriété manquante est une opportunité de citation IA perdue.

URLs relatives dans les champs URL

Tous les champs de type URL (url, image, logo, sameAs, hasMap) doivent contenir des URLs absolues complètes avec protocole https://. Une URL relative comme /assets/logo.png est invalide dans un JSON-LD.

Schema seulement sur la page d'accueil

Le Schema Dentist ne doit pas être limité à la page d'accueil. Il doit être présent sur toutes les pages du site — pages de services, pages de quartier, page Contact, page À propos. Chaque page est un point d'entrée potentiel pour une IA qui explore votre site.

L'installation du Schema Dentist est une intervention technique ponctuelle — en général 1 à 2 heures de travail pour un développeur, sur WordPress ou site custom. C'est un investissement unique : une fois le code en place, il fonctionne indéfiniment. Seules les mises à jour (horaires, nouveaux services, nouvelles langues) nécessitent des ajustements ponctuels. C'est l'un des meilleurs retours sur investissement en visibilité digitale pour un cabinet dentaire.

La logique est identique — c'est un bloc JSON-LD à injecter dans le <head> — mais la méthode diffère. Sur WordPress, utilisez RankMath (Local Business > Dentist), Yoast avec l'extension locale, ou ajoutez le code dans functions.php. Sur un site HTML statique, collez le JSON-LD directement dans le <head>. Sur React ou Next.js, injectez-le via next/head ou react-helmet.
